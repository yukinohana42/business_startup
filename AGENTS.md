# AGENTS

このプロジェクトでCodexやAIエージェントが作業するときのルールである。

## 必ず守る基準

- 最初に `02_SOURCE_OF_TRUTH.md` を読む。
- 次に `01_PROJECT_REQUIREMENTS.md` を読む。
- 作業前に project root、branch、HEAD、worktree status、remote URL を確認する。
- `02_SOURCE_OF_TRUTH.md` を最優先する。
- 中学2年生が読んで意味がわかる言葉にする。
- 親向け資料では、落ち着いた説明にする。
- 子供向け資料では、短い問い、身近な例、選べるワークを中心にする。
- 「起業しなければいけない」という押し付けをしない。
- 「会社員は危険」「サラリーマンは終わり」など、不安を煽る表現を避ける。
- 抽象論だけで終わらせず、ゲーム、部活、勉強、友達、スマホ、親、テスト、文化祭、お店など身近な例に戻す。
- docs-only作業と教材生成作業を分ける。
- 大規模変更は作業ブランチで行う。
- mainへの直接作業は、ユーザーが明示した場合に限る。
- ブラウザプロファイル、Cookie、Login Data、Cache、Crashpad dump、`.env`、秘密情報をcommitしない。
- 画像やキャラクターは、既存漫画、アニメ、ゲームの固有キャラ、ロゴ、固有の絵柄を再現しない。
- 各日にミッションを設定する。
- ただしミッション化によって学習目的を薄めない。
- 競争やランキングを安易に入れない。
- ゲーミフィケーションは `02_SOURCE_OF_TRUTH.md` と矛盾させない。
- 子供向け資料では「問い」「選択」「発見」「小さな挑戦」を入れる。
- 人物イラストを使う場合は完全オリジナルキャラクターにする。
- イラストは飾りではなく、価値や変化を伝えるために使う。

## 書き方

- 1文を短くする。
- 難しい言葉は、かならず言い換える。
- 「顧客」は「困っている具体的な人」と説明する。
- 「PMF」は「作ったものと欲しい人の気持ちがピタッとはまること」と説明する。
- 「失敗」は「赤信号。進む方向を変えるサイン」と説明する。
- 「価値」は「相手の状態が少し良くなること」と説明する。
- 「お金」は「価値が届いたことへの対価」と説明する。

## 各日の基本形

各日は次の構成を基本にする。

1. 目的
2. 今日わかること
3. 導入の問い
4. 身近な例
5. メインワーク
6. 発表または共有
7. まとめ
8. 宿題または次回までの観察

## 変更時の確認

教材を変更したら、次を確認する。

- `02_SOURCE_OF_TRUTH.md` と矛盾していないか。
- 子供向けと親向けのトーンが混ざっていないか。
- スライド、ワークシート、カード、台本の内容が同じ方向を向いているか。
- 次に直すことがあれば `TODO.md` に書いたか。
- docs-only commitに、教材生成物、画像生成物、PDF、PPTX、ブラウザプロファイルが混ざっていないか。

## Visual Learning System

子供向けのスライド、インフォグラフィック、カード、ワークシートを作る前に、`docs/visual_learning_system.md` を読む。

HTML/CSS、PNG、PDF、PPTX、A4印刷レイアウトへ進む前の停止条件:

- 子供がその画面で何をするかが定義されていない。
- キャラクターの行動がページごとに定義されていない。
- ゲーム要素が学習行動を進めていない。
- 親向け資料で親が採点者またはゲームマスターになっている。
- 単なる「見出し＋本文＋カード」の説明資料になっている。
- 丸アイコン、顔だけ、名前ラベルだけのキャラクターになっている。

この条件を満たさない場合は、生成を止め、設計修正案を出す。

## Gitとファイル管理

- `git add .` は使わない。
- 変更したファイルだけを明示的にstageする。
- 画像生成物、PDF、PPTX、HTML/CSS、Markdown文書を混ぜた大きなcommitにしない。
- 固定情報源を更新した場合は、ManifestとTODOも確認する。
